Deploy Simulator Architecture

CI/CD deployment simulator — system overview, components, and data flow

Overview

Deploy Simulator is a single-process Python application that simulates CI/CD deployments. It generates synthetic deployment records, stores them in memory, exposes a REST API via FastAPI, and serves a browser UI from the same server.

There is no separate frontend server and no persistent database. All state lives in memory for the lifetime of the server process.

Core Components

1. models.py — Data Template

Pydantic Deployment model defines the shape of every deployment record:

  • id — int, ≥ 1
  • service — billing-api, auth-service, notifications, frontend-web
  • status — success, fail, rolled-back
  • duration — float (seconds)
  • timestamp — ISO 8601 string
  • commit_sha — 7–40 character git SHA

2. deployment_ops.py — DeploymentOps

Central in-memory store. One instance per server session.

StructurePurpose
deployments (Queue)All deployments, FIFO order
_deployments_by_id (dict)O(1) lookup by id
service_dictPer-service queue + duration sum
anomaly_listDeployments flagged as anomalous
_status_cntrCount per status (success / fail / rolled-back)

Key methods:

  • add_deployment — enqueue, index, update counters, detect anomalies
  • read_deployments — list with optional service/status filters
  • read_deployment — fetch by id
  • p95_duration — 95th percentile duration per service
  • is_anomaly — duration > avg(last 5) + 2×SPREAD
  • success_rate — success count / total × 100
  • read_anomalies — return anomaly list

Protected by threading.RLock for safe access from the API and background thread.

3. dummy_generator.py — DummyGenerator

Produces random Deployment records on a background daemon thread.

  • create_dummy_deployment() — builds one record, auto-increments id
  • start_deployments(ops) — enqueue every 10 seconds
  • stop_deployments() — stop thread; id counter persists across start/stop
  • is_running() — whether the generator thread is active

Weighted random selection for services and statuses; frontend-web gets longer durations.

4. main.py — FastAPI Application

On startup (lifespan), creates one DeploymentOps and one DummyGenerator attached to app.state. All routes share these instances.

Structured error responses: { error, message, code, details }

5. Frontend — App/static/

  • P95 / Anomalies — collapsible sections, on-demand pull buttons
  • Deployment generation — toggle switch; syncs state via GET /deployments/status on load
  • Success Rate — live box, refreshes every 10s
  • Deployments table — auto-refreshes every 5s

Analytics Logic

P95 duration (per service)

  1. Copy deployments from service_dict[service].deployments
  2. If fewer than 5 → return {"message": "not enough deployments"}
  3. Sort by duration ascending
  4. Return item at index int(len × 0.95) − 1

Anomaly detection

On each add_deployment, before enqueueing:

  1. Take the latest 5 deployments for the same service
  2. Compute threshold = average(duration) + 2 × SPREAD (SPREAD = 5)
  3. If new duration > threshold → append to anomaly_list

Success rate

(status_cntr["success"] / sum(status_cntr.values())) × 100

Returns 0.0 when no deployments exist.

Session & Concurrency Model

  • Session scope — DeploymentOps and DummyGenerator live for the uvicorn process lifetime
  • Start/stop — same generator instance; id counter continues across stop/start
  • Thread safety — RLock on DeploymentOps; API handlers and generator thread share the store
  • Ephemeral — restarting the server clears all deployments and counters
